# What worked this week?

- Working on a separate branch for each issue worked well, as it allowed team members to work independently in parallel 
and reduced interference between tasks. This significantly improved development efficiency.

- Pull request reviews were consistently fast, which helped keep work moving forward without long delays and allowed 
issues to be merged in a timely manner.

- We scheduled two regular weekly meetings and also held an additional ad-hoc meeting to resolve a specific issue. 
This combination improved communication efficiency and helped unblock progress quickly when problems arose.

- Clarifying Javadoc requirements by asking Professor Skoteiniotis on Discord helped align the team’s understanding of 
documentation standards and reduced uncertainty during implementation.

- During the first weekly meeting, we broke the weekly goals into approximately 15 issues. This division of work was 
reasonable and made individual responsibilities clear, which helped the team stay organized.

- All three team members were supportive of each other’s work, actively participated in discussions, and responded to 
team messages promptly, which created a positive and productive team environment.

# What did not work this week?

- We encountered some minor merge conflicts because multiple team members worked on the same files simultaneously. 
To address this, we plan to better coordinate file ownership and communicate earlier when changes affect shared files.

- There was partial overlap between two tasks (PipelineValidationService and validators), which led to some 
duplicated effort. We resolved this by holding a temporary meeting to decide which code to keep, and we plan to define 
task boundaries more clearly in the future.

- The overall project structure was not fully clarified at the beginning of the week. As a result, some design 
decisions were made while coding rather than upfront. Although we eventually aligned on the structure through 
discussion, this slowed early development. In future sprints, we will finalize and document the project structure 
earlier before starting implementation.

- Coding style and Javadoc standards were not strictly defined before implementation began. This led to 
some rework later. After discussing the issue, we established clear standards, which should help prevent 
similar problems in future sprints.

- We initially enforced a relatively loose Checkstyle configuration and tightened it later in the week. This caused 
several CI failures after the stricter rules were introduced. Although we resolved the issues quickly by fixing 
Checkstyle violations in each team member’s code, defining and enforcing strict coding standards earlier would have 
reduced rework and CI disruptions.
